## 1. Planar Quadrotor

### What is "Planar"?

"Planar" means the drone is constrained to move in a 2D plane (x-y plane only).  
Think of it like a drone moving along a glass window — it can go left/right and up/down, but not forward/backward.

This is a **simplification** to make the math manageable before extending to full 3D.

### Degrees of Freedom (DoF)

> **DoF:** The number of independent ways a system can move. Equivalently, the minimum number of coordinates needed to fully specify the system's configuration.

- Planar quadrotor: **3 DoF** → `x` (horizontal), `y` (vertical), `θ` (roll angle)
- Each DoF can be controlled independently

### Step 1: 1D case (y-axis only)

Simplest case — drone only moves up and down:

$$m\ddot{y} = F_1 + F_2 - mg$$

$$\ddot{y} = \frac{u_1}{m} - g \quad \text{where } u_1 \triangleq F_1 + F_2$$

> **`≜` symbol:** Means "is defined as." It's a declaration — we're giving a name to something for convenience. Like declaring a variable in code: `u1 = F1 + F2`.

### Step 2: Full Planar Quadrotor (x, y, θ)

Applying Newton's second law in all directions:

$$\ddot{x} = -\frac{u_1}{m} \sin\theta \tag{x-direction}$$

$$\ddot{y} = \frac{u_1}{m} \cos\theta - g \tag{y-direction}$$

$$\ddot{\theta} = \frac{u_2}{I} \tag{rotation}$$

**Why sin/cos?**  
When the drone is tilted by angle θ, the thrust force is no longer pointing straight up — it splits into:
- Vertical component: `u₁ · cosθ` (still lifts the drone)
- Horizontal component: `u₁ · sinθ` (pushes the drone sideways)

This is why a drone must tilt to move horizontally.

**Sanity check:** When `θ = 0` (drone is level), `sinθ = 0` and `cosθ = 1`, so equation (y) reduces to the 1D case. ✓

## 2. Control Inputs

We don't directly command individual motor thrusts `F₁` and `F₂`. Instead, we use two combined inputs:

$$u_1 \triangleq F_1 + F_2 \quad \text{(Total Thrust — how much to lift?)}$$

$$u_2 \triangleq (F_2 - F_1) \times L \quad \text{(Total Torque — how much to tilt?)}$$

**Why combine them?**
- `u₁` controls vertical motion (go up/down)
- `u₂` controls rotation (tilt left/right)

This decomposition makes control design much cleaner.

**Intuition for Torque:**  
Think of a seesaw — if one side is heavier, it tips that way.  
If `F₂ > F₁`, the right motor pushes harder, so the drone tilts left.  
The longer the arm `L`, the more tilt you get from the same force difference.

> **Torque:** A "rotational force." Just as force causes linear acceleration (`F = ma`), torque causes angular acceleration (`τ = Iα`).

> **Thrust:** The upward force generated by a spinning propeller. A propeller pushes air downward; by Newton's third law, the drone gets pushed upward.

## 3. State-Space Representation

### Introducing State Variables

The **state** of a system is everything we need to know to predict its future motion.

We introduce velocity variables to convert 2nd-order ODEs into 1st-order:

$$v_x \triangleq \dot{x}, \quad v_y \triangleq \dot{y}, \quad \omega \triangleq \dot{\theta}$$

### 6 First-Order ODEs

$$\dot{x} = v_x$$
$$\dot{y} = v_y$$
$$\dot{\theta} = \omega$$
$$\dot{v}_x = -\frac{u_1}{m}\sin\theta$$
$$\dot{v}_y = \frac{u_1}{m}\cos\theta - g$$
$$\dot{\omega} = \frac{u_2}{I}$$

### Why convert to 1st-order?

Computers (numerical integrators like RK4) always assume **1st-order ODEs**.  
Converting makes simulation possible.

> **ODE (Ordinary Differential Equation):** An equation that contains derivatives. It describes *how a system changes over time*. Given the state at time `t=0`, we can solve the ODE to find the state at any future time.

### State Vector and Control Input Vector

$$\bar{x} = [x,\ y,\ \theta,\ \dot{x},\ \dot{y},\ \dot{\theta}]^T \quad \text{(6 states)}$$

$$\bar{u} = [u_1,\ u_2]^T \quad \text{(2 control inputs)}$$

> **`[...]ᵀ` (Transpose):** Written horizontally for space, but actually a column vector (vertical). The meaning is the same.  
> **Bar notation `x̄`:** Indicates a vector (multiple numbers grouped together), not a single scalar.

The full dynamics can then be written compactly as:

$$\dot{\bar{x}} = f(\bar{x},\ \bar{u})$$

"The rate of change of the state depends on the current state and the control inputs." This single equation captures everything.

### General Rule

| Quantity | Planar | 3D |
|----------|--------|-----|
| DoF | 3 | 6 |
| 2nd-order ODEs | 3 | 6 |
| States | 6 | 12 |
| 1st-order ODEs | 6 | 12 |

$$\text{DoF} = N \implies \text{States} = 2N \implies \text{1st-order ODEs} = 2N$$

## 4. Motor Model

In reality, we don't directly command thrust — we command **motor speed (RPM)**.

$$F = k_f \times (\text{RPM})^2$$

- `kf` = **thrust coefficient** — a physical constant unique to each drone/propeller.
- The relationship is **quadratic**: doubling RPM gives 4× the thrust.

**How to measure `kf`:**
1. Place the drone on a scale.
2. Sweep through different motor speeds.
3. Record the scale readings (convert kg → N).
4. The slope of the `F vs. RPM²` graph = `kf`.

> This is exactly what Assignment 1 asks you to do with real Crazyflie data.

**Three physical parameters** fully define our dynamics model:

| Parameter | Meaning | How to obtain |
|-----------|---------|---------------|
| `m` | mass | weigh the drone |
| `I` | moment of inertia | geometry or experiment |
| `kf` | thrust coefficient | motor sweep experiment (Assignment 1) |

> **Moment of Inertia `I`:** Rotational equivalent of mass. Measures how hard it is to spin something. Larger `I` → harder to rotate. Depends on how far mass is distributed from the rotation axis (e.g., a figure skater spins faster when arms are pulled in → `I` decreases).

## 5. 3D Quadrotor (Preview)

### Degrees of Freedom

Full 3D quadrotor has **6 DoF**:
- Position: `x, y, z` (3 numbers)
- Orientation: `φ, θ, ψ` (3 Euler angles)

### Euler Angles

To describe 3D orientation, we use **Euler angles** — three rotations applied sequentially.

We define two coordinate frames:
- **Inertial frame (I):** Fixed to the ground. `ēx` (East), `ēy` (North), `ēz` (Up)
- **Body frame (B):** Attached to the drone. `b̄x` (forward), `b̄y` (left), `b̄z` (up)

**Space 1-2-3 convention** (used in this course):

| Step | Rotation | Axis | Angle | Effect |
|------|----------|------|-------|--------|
| 1 | Rotate B about `ēx` | x-axis | φ (phi) | **Roll** — tilt left/right |
| 2 | Rotate B about `ēy` | y-axis | θ (theta) | **Pitch** — tilt forward/backward |
| 3 | Rotate B about `ēz` | z-axis | ψ (psi) | **Yaw** — spin in place |

**Axis = rotation axis intuition:**  
Imagine skewering the drone with a stick along that axis. The drone rotates *around* the stick.
- x-axis stick (front-to-back) → drone tilts left/right = Roll
- y-axis stick (left-to-right) → drone tilts forward/backward = Pitch
- z-axis stick (bottom-to-top) → drone spins in place = Yaw

**Right-Hand Rule:**  
Point your right thumb along the rotation axis. The direction your fingers curl = positive rotation direction.

> ⚠️ **Convention warning:** There are many different Euler angle conventions (different orders, different axes). Using the wrong convention is a common source of bugs. Always confirm which convention you and your teammates are using!

### 3D State Vector (12 states)

$$\bar{x} = [x,\ y,\ z,\ \phi,\ \theta,\ \psi,\ \dot{x},\ \dot{y},\ \dot{z},\ p,\ q,\ r]^T$$

where `p, q, r` are angular velocities (slightly different from `φ̇, θ̇, ψ̇` — this distinction matters in 3D).

### 3D Control Inputs (4 inputs)

$$\bar{u} = [F_{tot},\ M_1,\ M_2,\ M_3]^T$$

| Input | Meaning | Controls |
|-------|---------|---------|
| `Ftot` | Total thrust (sum of 4 motors) | altitude |
| `M₁` | Moment about x-axis | Roll |
| `M₂` | Moment about y-axis | Pitch |
| `M₃` | Moment about z-axis | Yaw |

## Summary

```
Goal: Make quadrotor hover
  ↓
Step 1 (Today): Dynamics — model how the drone moves
  → EOM: ẍ = -(u₁/m)sinθ,  ÿ = (u₁/m)cosθ - g,  θ̈ = u₂/I
  → State: x̄ = [x, y, θ, ẋ, ẏ, θ̇]ᵀ  (6 states)
  → Input: ū = [u₁, u₂]ᵀ  (2 inputs)
  → Compact form: x̄̇ = f(x̄, ū)

Step 2 (Next 3 lectures): Control — design u₁, u₂ to keep drone at hover
```

**Hover condition derivation:**

Hover = drone is completely stationary in the air.
This means all accelerations must be zero: `ẍ = ÿ = θ̈ = 0`, and the drone must be level: `θ = 0`.

Substitute these conditions into the three EoMs:

**Equation 1 (x-direction):**
```
0 = -(u₁/m) · sin(0) = 0  ✓
```
Automatically satisfied since `sin(0) = 0`. No constraint on `u₁` from this equation.

**Equation 2 (y-direction):**
```
0 = (u₁/m) · cos(0) - g
0 = (u₁/m) · 1 - g
∴ u₁ = mg
```
Thrust must exactly cancel gravity.

**Equation 3 (rotation):**
```
0 = u₂ / I
∴ u₂ = 0
```
No torque needed — drone stays level.

**Result:**
- `u₁ = mg` — thrust equals the drone's weight
- `u₂ = 0` — no rotational input
- `θ = 0` — drone remains horizontal

This makes intuitive sense: to hover, you only need to push up exactly as hard as gravity pulls down, with no tilting.
